## 1. The Auto-Pathfinding Problem | 自动寻路问题

When a developer says "add a purchase order approval workflow", the AI must answer:

1. **Which project line?** → Mainline development
2. **Which rules apply?** → AGENTS.md, project_rules.md, database rules
3. **Which skill leads?** → Depends: backend-first if new API, frontend-first if UI change
4. **What to check first?** → Existing purchase_order table, existing approval flow, current permission setup
5. **Execution order?** → DB migration → Entity → Service → Controller → Frontend → Integration test

**This is the auto-pathfinding problem**: given a natural language request, determine the optimal execution path through the skill system.

## 2. The Routing Algorithm | 路由算法

The i-rule-dispatcher implements a deterministic routing algorithm:

`
INPUT: User request (natural language)

STEP 1: Classify Task Line
  - Contains "fix"/"bug"? → mainline
  - Contains "audit"/"check"/"review"? → audit
  - Contains "competitor"/"research"/"market"? → research  
  - Contains "deploy"/"release"? → release
  - Contains "architecture"/"boundary"/"domain"? → governance
  - Default: mainline

STEP 2: Classify Task Type
  - Cross-module (frontend + backend + DB)? → ai-task-decomposer
  - Architecture decision needed? → ai-architect-governor
  - Project scheduling? → ai-chief-planner
  - Command execution? → ai-command-executor
  - Otherwise: direct to execution layer

STEP 3: Match Tech Stack
  - Keywords "vue"/"component"/"page"? → vue
  - Keywords "spring"/"java"/"api"? → java-springboot
  - Keywords "sql"/"table"/"schema"? → mysql-best-practices
  - Keywords "docker"/"container"? → docker-expert

STEP 4: Select First Documents
  - Always: rules/AGENTS.md
  - If project_rules.md exists: load it
  - Lead skill's SKILL.md
  - Relevant docs/ category

STEP 5: Define Pre-Checks
  - DB changes → SHOW TABLES, DESCRIBE
  - Code changes → file existence, grep patterns
  - Runtime → port check, health endpoint

OUTPUT: Routing result with lead skill, support skills, docs, checks
`

## 3. Task Line Priority System | 任务线优先级

Multiple task lines can be active simultaneously:

| Line | Priority | Typical Owner | Max Parallel |
|---|---|---|---|
| **Mainline** | Highest | Primary developer | Active development |
| **Governance** | High | Architect | Architecture decisions |
| **Audit** | Medium | QA / Reviewer | Quality checks |
| **Research** | Low | Anyone | Investigation |
| **Release** | Highest (during release window) | DevOps | Deployment |

### Line Conflict Resolution

When tasks from multiple lines compete for the same resource:
1. **Release line** takes absolute priority during release windows
2. **Mainline blocker fixes** take priority over new feature work
3. **Governance decisions** that block mainline get expedited
4. **Research** is always preemptible

## 4. Dependency-Aware Scheduling | 依赖感知调度

The i-task-decomposer produces a dependency graph:

`mermaid
graph TD
    A[B-01: DB Migration] --> B[B-02: Entity Layer]
    A --> C[B-03: Test Data]
    B --> D[B-04: Service Layer]
    D --> E[B-05: Controller Layer]
    E --> F[B-06: Frontend Service]
    F --> G[B-07: Frontend Page]
    C --> H[B-08: Integration Test]
    E --> H
    G --> H
`

### Scheduling Rules

| Rule | Description |
|---|---|
| **Blocker-first** | Tasks that unblock others execute first |
| **No hot-file collision** | Tasks writing to the same file cannot run in parallel |
| **Frontend waits for backend API** | Frontend dev starts only after API endpoints are testable |
| **Tests wait for implementation** | Integration tests start after both frontend and backend complete |

## 5. Routing Examples | 路由示例

### Example 1: "Fix the product search that returns 500"

`
Classification:
  Line: mainline (bug fix)
  Type: execution (specific file/API)
  Stack: depends on project tech
  
Routing:
  Lead: java-springboot (if backend) or vue (if frontend)
  First check: server logs, /actuator/health, error stack trace
  Docs: rules/AGENTS.md
  Pre-check: curl the failing endpoint, check error log
`

### Example 2: "Add inventory valuation report"

`
Classification:
  Line: mainline (feature)
  Type: decomposition (new report = DB + backend + frontend)
  Stack: java-springboot + vue
  
Routing:
  Lead: ai-task-decomposer (break into batches)
  First docs: rules/AGENTS.md, existing report examples
  Pre-checks: inventory tables, existing report patterns, permission setup
  
Decomposition output:
  B-01: DB — add report query tables
  B-02: Backend — report service + controller
  B-03: Frontend — report page + filters
  B-04: Integration test
`

### Example 3: "Audit all purchase flows for closure"

`
Classification:
  Line: audit
  Type: governance (quality audit)
  
Routing:
  Lead: ai-flow-closure-audit
  Support: ai-domain-boundary-mapper
  First docs: rules/AGENTS.md, existing business process docs
  Pre-checks: purchase module codebase, API endpoints, DB schema
  
Output: closure audit report with prioritized gaps
`

## 6. When Auto-Pathfinding Fails | 自动寻路失败时

The dispatcher has an **escalation path**:

1. **Ambiguous request** → Ask the user one clarifying question (not multiple)
2. **Unknown domain** → Route to i-chief-planner for research-based scheduling
3. **Conflict detected** → Flag the conflict, defer to human decision
4. **No matching skill** → Route to i-skill-evolver to create a skill candidate
