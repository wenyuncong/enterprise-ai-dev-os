## 1. Overview | 概述

This guide explains how to **bootstrap this methodology for any project** — new or existing, any tech stack, any domain.

The methodology is a **zero-dependency file system**, not a SaaS product. You copy folders, select skills, and the AI tool reads them at session start. No installation, no API keys, no runtime.

---

## 2. Prerequisites | 前置条件

| Requirement | Minimum |
|---|---|

| Version control | Git (or equivalent) |

## 3. Quick Start: New Project | 新项目快速开始


### Step 0: Classify Your Project
Before copying any files, run the classification:

1. Determine: Brownfield or Greenfield?
2. Determine: Rapid Prototype, AI-Native, or Enterprise?
3. Determine: Web only, Web+Mobile, Web+WeChat, or All platforms?
4. Determine: Monolith, Atomic Services, or Microservices?

This classification determines which skills to copy and which architecture to apply. See `ai-project-classifier` for the full decision framework.

## 4. Migration: Existing Project | 已有项目迁移

### Phase 1: Audit (Day 1)
1. Copy `rules/AGENTS.md` to project root
2. Run `ai-rule-dispatcher` for full project audit
3. AI will produce a gap analysis report:
   - Which methodology rules are already followed?
   - Which are missing?
   - What is the risk of introducing methodology changes?

### Phase 2: Core Skills (Week 1)
1. Copy `skills/core/` to project
2. Start using `ai-rule-dispatcher` for task routing (low risk — it only advises, doesn't change code)
3. Start using `ai-task-decomposer` for new complex tasks (doesn't affect existing code)
4. Start daily writebacks under `docs/每日调研回写/`

### Phase 3: Governance (Week 2)
1. Copy `skills/governance/` to project
2. Run `ai-flow-closure-audit` on one critical module
3. Run `ai-frontend-audit` on one key page set
4. Address findings incrementally

### Phase 4: Full Adoption (Week 3+)
1. Enforce 13-step order for all new development
2. Activate `ai-skill-evolver` for continuous improvement
3. Integrate deployment gates
4. Train team members on methodology usage

## 5. Customization | 定制化

### Adding a New Tech Skill
```bash
# Create the skill directory
mkdir -p {your-project}/skills/tech/{your-stack}

# Write SKILL.md following this template:
#   - Purpose
#   - Core Rule
#   - When to Use
#   - Standard Workflow
#   - Guardrails
#   - Project-Specific Notes (YOUR custom rules go here)
```

### Modifying Core Methodology Rules
1. Never modify `skills/core/` skills directly — they are the canonical versions
2. Instead, create project-specific overrides in `rules/project_rules.md`
3. Project rules take precedence over core rules

### Adding Domain-Specific Governance Skills
1. Create new skill in `skills/governance/`
2. Name it `ai-{domain}-audit` for audit skills
3. Follow the governance skill template (purpose → core rule → workflow → guardrails)

## 7. Troubleshooting | 常见问题

| Problem | Solution |
|---|---|
| "AI ignores the rules" | Ensure `rules/AGENTS.md` is in the project root with correct trigger config |
| "Skills conflict with each other" | Run `ai-rule-dispatcher` — it selects ONE lead skill |
| "Documentation is growing too fast" | Only write back findings that affect future AI sessions; skip trivial notes |
| "Skills are outdated" | Run `ai-skill-evolver` — it will detect and fix rot |
| "Team doesn't know how to use it" | Have each team member start one session with "Read rules/AGENTS.md" |
