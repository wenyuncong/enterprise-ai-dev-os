## 1. Why Release Governance? | 为什么需要发布治理？

Step 13 (Runtime Verify) proves the page renders. But it doesn't prove:
- All business processes work end-to-end
- All P0 bugs are fixed
- All features are actually complete (not just "code written")
- The version boundary is clear (what's in, what's out)
- Manual testing has been done on critical paths

GERP's release closure system caught problems that automated checks missed. This methodology document formalizes that pattern.

## 4. P0/P1/P2 Bug Classification | 缺陷分级

### P0 — Blocks Release (发布阻断)
**Must be zero before release.**

| Criteria | Example |
|---|---|
| Core business flow broken | Cannot create purchase order |
| Data loss or corruption | Transaction deletes wrong records |
| Security vulnerability | Unauthenticated API access |
| Tenant isolation breach | Tenant A sees Tenant B's data |
| Payment/billing error | Wrong amount charged |

### P1 — Must Fix (必须修复)
**< 5 allowed at release. Each documented with workaround.**

| Criteria | Example |
|---|---|
| Important feature broken | Export fails for > 1000 rows |
| UX blocking issue | Modal cannot be closed on mobile |
| Performance degradation | Page load > 10 seconds |
| Data display error | Report shows wrong totals |

### P2 — Nice to Fix (建议修复)
**Can ship with P2 bugs. Documented in known issues.**

| Criteria | Example |
|---|---|
| Cosmetic issue | Misaligned button on one page |
| Edge case | Special character in name breaks display |
| Optimization | Redundant API call on page load |

## 5. Manual Test Ledger | 人工测试台账

Automated tests can't verify everything. Critical paths need human eyes.

### Template
```markdown
## 人工测试台账

| # | 测试场景 | 模块 | 操作步骤 | 预期结果 | 实际结果 | 测试人 | 日期 |
|---|---|---|---|---|---|---|---|
| 1 | 创建采购订单-完整流程 | 采购 | 新建→添加行→保存→提交→审批 | 状态流转正确 | ✅ | | |
| 2 | 库存入库-批次管理 | 仓库 | 收货→质检→上架 | 库存更新正确 | ⚠️ 批次号未自动生成 | | |
```

## 6. Business Process Acceptance Matrix | 业务流程验收矩阵

Every key business process must be verified end-to-end.

### Template
```markdown
## 业务流程验收矩阵

| 流程 | 模块 | Page→API→DB→Writeback | 自动测试 | 人工验证 | 状态 |
|---|---|---|---|---|---|
| 采购订单→入库→付款 | 采购+仓库+资金 | ✅ | ✅ | ✅ | PASS |
| 销售订单→出库→收款 | 销售+仓库+资金 | ✅ | ✅ | ⚠️ | P1: 收款未自动核销 |
| 生产工单→领料→完工 | 生产+仓库 | ✅ | ❌ | — | BLOCKED: 无测试 |
```

## 7. Multi-Conversation Coordination | 多对话协作

GERP release closure involved multiple AI conversations and human testers. Coordination rules:

### Rules
1. **One master control document** — single source of truth for release status
2. **Conversation naming**: `{date}_{role}_{task}` — e.g., `2026-06-12_tester_regression`
3. **No silent overrides** — if two conversations modify the same file, flag it
4. **Daily sync** — each conversation updates status in master control at end of day
5. **Lock protocol** — once locked, only P0 fixes allowed; all changes require approval

## 8. Integration with 13-Step Methodology | 与13步集成

```
Step 12: Schema Sync
Step 13: Runtime Verify
    │
    ▼
Step 13a: Release Governance (NEW — for enterprise releases)
    │
    ├── Create release folder under docs/发布闭环/
    ├── Populate lock-version master control
    ├── Run P0/P1/P2 bug audit
    ├── Execute manual test ledger
    ├── Verify business process acceptance matrix
    └── Gate: All P0 = 0, P1 < 5, Manual tests pass
    │
    ▼
RELEASE
```

**Rule**: Steps 13a is MANDATORY for Enterprise (Archetype C) releases. Optional for AI-Native (Archetype B). Skip for Rapid Prototype (Archetype A).
