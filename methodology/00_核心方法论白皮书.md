## Executive Summary | 概述

This methodology defines a **complete, self-evolving system for AI-assisted enterprise software development**. It covers the full lifecycle:

`
Requirements  ->  Plan  ->  Task Decomposition  ->  Auto-Pathfinding
     ->  Code Execution  ->  Test Writing  ->  MD Documentation Writeback
     ->  Skill Auto-Evolution
`

Unlike "vibe coding" — where developers loosely prompt AI and hope for good output — this methodology introduces **engineering discipline into AI-assisted development**: mandatory check-before-execute, 13-step development order, data-driven truth sources, and a self-evolving skill system that improves with every completed task.

### Why This Exists | 为什么需要这套方法论

The AI coding tool market is exploding — Cursor, Codex, Copilot, Windsurf — but the market has a gaping hole:

| What exists | What's missing |
|---|---|

This methodology fills that gap. It was not designed in a vacuum — it was forged in the fire of building a real multi-tenant ERP system (purchase, sales, inventory, finance, production, CRM, WMS, TMS) across 42+ specialized AI skills and 2000+ documentation artifacts.

## 1. The Core Engine | 核心引擎

The methodology is powered by a **four-layer skill architecture**:

### Layer 1: Routing | 路由层
**Skill**: i-rule-dispatcher

Before any task begins, this layer answers:
- Which project line does this task belong to?
- Which rules and documents must be read first?
- Which skill should lead execution?
- What factual checks (DB, code, runtime) must happen first?

### Layer 2: Decomposition | 分解层
**Skill**: i-task-decomposer

Complex tasks are decomposed into safe, executable batches:
- Scope freeze (in/out of scope)
- Current-state verification vs target design vs implementation
- Dependency-aware ordering
- Acceptance criteria and evidence rules per batch

### Layer 3: Execution | 执行层
**Skills**: Domain-specific (backend-dev, frontend-dev, database-manager, etc.)

The execution layer follows a **mandatory 13-step development order**:

| # | Backend | Frontend |
|---|---|---|
| 1 | Database Init & Verification | — |
| 2 | Entity Layer | — |
| 3 | Mapper/Repository Layer | — |
| 4 | Service Layer (business logic) | — |
| 5 | Controller/API Layer | — |
| 6 | Build & Compile Verification | — |
| 7 | — | TypeScript Interface Generation |
| 8 | — | Service/API Layer |
| 9 | — | Vue/React Component Development |
| 10 | Integration Testing | Integration Testing |
| 11 | Seed Data & Validation | UI Smoke Testing |
| 12 | Schema Sync & Release | Build & Deploy |

Each step **must be verified** before moving to the next. No skipping.

### Layer 4: Evolution | 进化层
**Skill**: i-skill-evolver

After every task completion, the evolver:
1. Detects repeated patterns, mistakes, or missing context
2. Classifies the gap (skill update, new skill, doc update, rule change)
3. Updates the relevant SKILL.md or creates a new one
4. Records the evolution in the skill archive
5. Syncs the change to the active AI tool (Codex, Cursor, etc.)

## 2. Engineering Discipline | 工程纪律

These are the **non-negotiable rules** that separate this methodology from vibe coding:

### 2.1 Check Before Execute | 先检查后执行
**Never assume. Always verify.**
- Before DB changes: SHOW TABLES; DESCRIBE table; SELECT COUNT(*)
- Before code changes: Verify file existence, grep for existing patterns
- Before runtime changes: Check ports, processes, logs

### 2.2 No Degradation | 禁止降级方案
**Find the root cause. Fix it completely.**
- No temporary workarounds
- No "just make it work" patches
- Every fix must address the underlying issue

### 2.3 Data-Driven Development | 数据驱动开发
**The database is the single source of truth.**
- All structural decisions start from DB verification
- Code and documentation follow the DB, not the other way around
- Schema changes go through formal migration scripts

### 2.4 Zero-Fluff UI | 零废话界面
**ERP/SaaS/admin pages contain zero decorative text.**
- No welcome messages, marketing copy, or placeholder help text
- Only actionable prompts: field constraints, validation rules, business risk warnings
- Tooltips and inline validation, never standalone explanation blocks

### 2.5 Plan-Driven Execution | 计划驱动
**Every task must have a plan before execution.**
- Task analysis, complexity assessment, skill matching
- Step-by-step plan with acceptance criteria per step
- PDCA loop: Plan -> Do -> Check -> Act

## 3. Documentation System | 文档体系

The methodology includes a **self-growing documentation system** organized into six categories:

| Category | Purpose | Update Frequency |
|---|---|---|
| **Architecture Decision Records** | Cross-domain architecture decisions | Per decision |
| **Daily Investigation Writebacks** | Automated daily state reports | Daily |
| **Business Process Documentation** | End-to-end business flow docs | Per flow change |
| **Deployment & Operations** | Deployment guides, rollback procedures | Per release |
| **Testing & Acceptance** | Test plans, regression records | Per test cycle |
| **Master Project Control** | Overall project status, task boards | Continuous |

The documentation system is **not documentation for documentation's sake** — it is the **memory layer** that allows AI agents to maintain context across sessions and enables new agents to onboard instantly.

## 4. Skill Auto-Evolution | Skill 自动进化

This is the methodology's most unique capability. Unlike static rule files, the skill system **improves with use**:

`
Task Execution -> Pattern Detection -> Gap Classification
     -> Skill Update/Creation -> Archive Record -> Next Task Benefits
`

### Evolution Triggers
- Repeated task patterns across modules
- Repeated mistakes or missing-context moments
- Stable output templates that should be formalized
- Project path or rule changes
- User feedback that a skill is not effective

### Evolution Actions
- Update existing SKILL.md with new patterns
- Create new skill when a pattern crosses 2+ task types
- Update skill maturity rating (declared -> callable -> effective -> evolving)
- Record evolution evidence in skill archive
