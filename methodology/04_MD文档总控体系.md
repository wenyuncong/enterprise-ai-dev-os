## 1. Why Documentation Matters in AI Development | 为什么AI开发需要文档

AI agents have a fundamental limitation: **no persistent memory across sessions.** Each new session starts with zero context about what happened before.

Without a documentation system:
- AI repeats the same investigations
- AI makes the same mistakes
- AI cannot reference past decisions
- Every session is "starting from scratch"

The MD documentation system is the **memory layer** that bridges AI sessions.

## 2. The Six-Category System | 六类文档体系

`
docs/
  _templates/                    ← Templates for new projects
  |
  架构决策记录/                   ← Category 1: Architecture Decision Records
  每日调研回写/                   ← Category 2: Daily Investigation Writebacks
  业务流程全案/                   ← Category 3: Business Process Documentation
  部署运维手册/                   ← Category 4: Deployment & Operations
  测试验收报告/                   ← Category 5: Testing & Acceptance
  全项目总控/                     ← Category 6: Master Project Control Hub
`

### Category 1: Architecture Decision Records (ADR)

**Purpose**: Record WHY architecture decisions were made.

**Template**: ADR format with context, decision, alternatives, consequences.

**Update frequency**: Per decision.

**Example entry**:
`
ADR-001: Multi-Tenant Schema Isolation
Status: Accepted
Decision: Each tenant gets a separate database schema
Alternatives: Shared tables with tenant_id column (rejected: security risk)
Consequences: Schema sync complexity, but full data isolation
`

### Category 2: Daily Investigation Writebacks

**Purpose**: Automatically record daily findings, decisions, and skill evolution triggers.

**Update frequency**: Every development session.

**Example entry**:
`
2026-06-17 — Purchase Order Save Performance
Finding: purchase_order table missing index on tenant_id + order_date
Action: Added composite index, query time dropped from 2.3s to 0.1s
Skill affected: mysql-best-practices — add index optimization pattern
`

### Category 3: Business Process Documentation

**Purpose**: Document end-to-end business flows with all layers verified.

**Update frequency**: Per process change.

**Content**: Process flow diagram, step-by-step details, related entities, parameters, writeback paths.

### Category 4: Deployment & Operations

**Purpose**: Deployment guides, rollback procedures, environment configuration.

**Update frequency**: Per release.

**Content**: Server setup, build commands, environment variables, health checks, rollback steps.

### Category 5: Testing & Acceptance

**Purpose**: Test plans, test cases, regression records, acceptance sign-offs.

**Update frequency**: Per test cycle.

**Content**: Test case matrix, pass/fail results, bug tracking, acceptance decisions.

### Category 6: Master Project Control Hub

**Purpose**: Overall project status, task boards, milestone tracking, skill maturity matrix.

**Update frequency**: Continuous.

**Content**: Master task board, milestone progress, skill inventory, team assignments.

## 3. The Writeback Loop | 文档回写循环

Documentation is not a separate phase — it's embedded in the execution cycle:

`
Task Execution
    |
    v
Writeback Decision: Does this task produce a reusable finding?
    |
    +-- YES --> Write to the relevant doc category
    |            |
    |            v
    |           ai-skill-evolver triggered (if pattern detected)
    |
    +-- NO  --> No writeback needed (trivial task)
`

### What Gets Written Back

| Finding | Category |
|---|---|
| Architecture decision | 架构决策记录 |
| Investigation result | 每日调研回写 |
| Business flow change | 业务流程全案 |
| Deployment config change | 部署运维手册 |
| Test result | 测试验收报告 |
| Project status change | 全项目总控 |

### What Does NOT Get Written Back
- Trivial bug fixes (typo, color change)
- Single-line code changes
- Routine dependency updates
- Already-documented findings

## 4. Documentation as AI Memory | 文档即AI记忆

When a new AI session starts, it reads documentation in this order:

`
1. rules/AGENTS.md             ← Core rules
2. docs/全项目总控/             ← Current project state
3. docs/每日调研回写/ (last 3)  ← Recent context
4. docs/架构决策记录/            ← Architecture decisions
5. Task-specific docs           ← Relevant to current task
`

This gives the AI the equivalent of "reading the team's Slack history for the past week" — in under 30 seconds.

---

## 5. The "Golden Rule" of AI Documentation | AI文档金科玉律

> **"Write documentation for the NEXT AI session, not for human posterity."**

Traditional documentation is written for humans to read months later. AI documentation is written for the **next AI session that starts in a few hours**. This means:

- **Concise, not verbose** — AI reads fast, but context windows are limited
- **Structured, not narrative** — Tables and checklists over prose
- **Verifiable, not speculative** — Cite file paths, line numbers, DB tables
- **Actionable, not descriptive** — Every entry should drive a decision or action

---

## 6. Anti-Patterns | 反模式

| Anti-Pattern | Why It Fails |
|---|---|
| **Documenting everything** | Clutters context, AI can't find signal in noise |
| **Documenting only for humans** | Prose-heavy docs waste AI context window |
| **No daily writebacks** | Each AI session starts from zero |
| **Docs in random locations** | AI can't discover relevant context |
| **Stale docs** | AI trusts docs over code, makes wrong decisions |
