## 1. Why a Fixed Order? | 为什么需要固定顺序？

In traditional development, the order of operations is implicit. In AI-assisted development, **explicit ordering prevents three classes of errors**:

| Error Class | Example | Prevention |
|---|---|---|
| **Frontend-first trap** | Building a UI before the API exists | Enforced: frontend starts at Step 7 |
| **Skip-verification trap** | Writing code without checking DB state | Enforced: every step has mandatory verification |
| **Assumption drift** | AI assumes a table exists that was never created | Enforced: Step 1 is always DB verification |

## 2. The 13 Steps | 完整13步

### Backend Phase | 后端阶段

#### Step 1: Database Initialization & Verification
`
Actions:
  1. Check: SHOW TABLES; DESCRIBE target_table;
  2. Write SQL migration script (if new/modified tables)
  3. Execute script
  4. Verify: SHOW TABLES; DESCRIBE target_table; SELECT COUNT(*);

Artifacts: database/migrations/V{NNN}__description.sql
Gate: All tables/columns exist and match specification
`

#### Step 2: Entity / Model Layer
`
Actions:
  1. Generate entity class from DB schema
  2. Add comments to every field
  3. Add validation annotations
  4. Verify: field names match DB column names exactly

Artifacts: {module}/entity/{Entity}.java (or equivalent)
Gate: Every DB column has a corresponding entity field with comment
`

#### Step 3: Mapper / Repository Layer
`
Actions:
  1. Create mapper interface extending BaseMapper/Repository
  2. Add custom query methods with XML/annotation SQL
  3. Verify: execute each custom query, check result correctness

Artifacts: {module}/mapper/{Entity}Mapper.java + XML
Gate: All custom queries return expected results
`

#### Step 4: Service Layer
`
Actions:
  1. Implement business logic in service class
  2. Add transaction management (@Transactional)
  3. Add cache configuration if needed
  4. Verify: write and run unit tests

Artifacts: {module}/service/{Entity}Service.java + tests
Gate: All unit tests pass
`

#### Step 5: Controller / API Layer
`
Actions:
  1. Create REST controller with standard CRUD endpoints
  2. Add API documentation (Swagger/OpenAPI annotations)
  3. Add permission annotations
  4. Verify: curl each endpoint, check response format

Artifacts: {module}/controller/{Entity}Controller.java
Gate: All endpoints return correct HTTP status and data format
`

#### Step 6: Build & Compile Verification
`
Actions:
  1. Clean compile: mvn clean compile (or equivalent)
  2. Start service locally
  3. Check health endpoint: /actuator/health
  4. Smoke test: one API call per new endpoint

Artifacts: Build log, health check response
Gate: Build succeeds, service starts, health is UP
`

### Frontend Phase | 前端阶段

#### Step 7: TypeScript Interface Generation
`
Actions:
  1. Generate TypeScript interfaces from backend DTOs/Entities
  2. Verify: every field matches backend, types are correct
  3. Add JSDoc comments

Artifacts: {app}/src/types/{Entity}.ts
Gate: Interface fields 1:1 match backend DTO
`

#### Step 8: Frontend Service / API Layer
`
Actions:
  1. Create API service with typed methods
  2. Add error handling (network errors, business errors)
  3. Add request/response interceptors if needed
  4. Verify: call each method, check network tab

Artifacts: {app}/src/services/{Entity}Service.ts
Gate: All API calls succeed with correct response types
`

#### Step 9: Frontend Component / Page
`
Actions:
  1. Create Vue/React component
  2. Follow project component conventions
  3. Implement loading, empty, error states
  4. Verify: open page in browser, test all interactions

Artifacts: {app}/src/views/{Entity}/index.vue (or equivalent)
Gate: Page renders correctly, all CRUD operations work
`

### Integration Phase | 联调阶段

#### Step 10: Integration Testing
`
Actions:
  1. End-to-end workflow test (create -> read -> update -> delete)
  2. Edge case testing (invalid input, concurrent operations)
  3. Error handling test (backend down, network error)
  4. Verify: all test cases pass

Artifacts: Test report
Gate: All integration test cases pass
`

#### Step 11: Seed Data & Validation
`
Actions:
  1. Write test data scripts
  2. Insert into test database
  3. Verify data in UI (every field renders correctly)
  4. Edge cases: null values, max-length strings, special characters

Artifacts: database/seed/{module}_test_data.sql
Gate: All seed data displays correctly in UI
`

#### Step 12: Schema Sync & Release
`
Actions:
  1. Compare template schema with all instance schemas
  2. Generate sync scripts for any drift
  3. Execute sync
  4. Verify: all schemas match template

Artifacts: database/sync/{date}_schema_sync.sql
Gate: Zero schema drift across all instances
`

## 3. Verification Checklist | 验证清单

Every step has a **binary gate**: pass or block. No "mostly done."

`
[ ] Step 1:  DB tables exist and match spec          ─── BLOCK if fail
[ ] Step 2:  Entity fields match DB columns           ─── BLOCK if fail
[ ] Step 3:  Custom queries return expected results   ─── BLOCK if fail
[ ] Step 4:  Unit tests pass                          ─── BLOCK if fail
[ ] Step 5:  API endpoints return correct responses   ─── BLOCK if fail
[ ] Step 6:  Build succeeds, service healthy          ─── BLOCK if fail
[ ] Step 7:  TS interfaces match backend DTOs         ─── BLOCK if fail
[ ] Step 8:  API service calls succeed                ─── BLOCK if fail
[ ] Step 9:  Page renders, CRUD works                 ─── BLOCK if fail
[ ] Step 10: Integration tests pass                   ─── BLOCK if fail
[ ] Step 11: Seed data displays correctly             ─── BLOCK if fail
[ ] Step 12: All schemas synced                       ─── BLOCK if fail
[ ] Step 13: Runtime verify passes                    ─── BLOCK if fail
`

---


#### Step 13: Browser Runtime Verification

`
Actions:
  1. Verify backend server is running on target port
  2. Run: node skills/governance/ai-runtime-verify/scripts/verify.js --url {APP_URL}
  3. Check output: "passed" must be true
  4. If P0 checks fail → fix errors and re-run Step 13
  5. If P1 warnings → note and decide whether to fix before release

Artifacts: skills/governance/ai-runtime-verify/screenshots/verify-{timestamp}.png
Gate: All 6 P0 checks must pass:
  - No console errors
  - Loading overlay removed properly
  - Core DOM elements render with content
  - All API calls return 2xx/3xx
  - No white screen / blank page
  - No infinite refresh loop

⚠️ CRITICAL RULE:
  An AI agent MUST NOT declare "development complete" 
  unless ai-runtime-verify returns "passed": true.
`

## 4. Exceptions & Shortcuts | 例外与捷径

Shortcuts are **explicitly allowed** in these cases:

| Scenario | Allowed Shortcut |
|---|---|
| Pure UI fix (no API/DB change) | Skip Steps 1-6, start at Step 7. Step 13 always required. |
| Pure DB migration | Only Steps 1 + 12 |
| Documentation only | Skip all steps, go directly to docs |
| Configuration change | Steps 1 (verify) + 6 (compile) + 12 (sync) |

**Rule**: Any shortcut must be explicitly stated with justification before execution begins.

---

## 5. Common Violations & Consequences | 常见违规与后果

| Violation | Consequence |
|---|---|
| Skipping DB check (Step 1) | Code written against wrong schema → 500 errors |
| Writing frontend before backend API (Step 7 before 5) | UI built against imaginary API → rework |
| Skipping build verification (Step 6) | Compile errors discovered late → cascading fixes |
| No integration test (Step 10) | Regressions discovered in production |
| Schema sync skipped (Step 12) | Multi-tenant drift → production failures |
| Runtime verify skipped (Step 13) | Page loads with JS errors, broken UX → AI claims done but page doesn't work |
