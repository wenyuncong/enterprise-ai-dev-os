# 08 — Project Folder Structure Standard | 项目文件夹结构标准

> **Part of**: Enterprise-Grade Full AI Development Methodology
> **Prerequisites**: 00_核心方法论白皮书, ai-project-classifier

---

## 1. Why a Standard Structure? | 为什么需要标准结构？

GERP project analysis revealed:

| Problem | Root Cause | Fix |
|---|---|---|
| 55 loose files in project root | No root-level discipline | Only AGENTS.md + .editorconfig at root |
| ~300 loose SQL files in database/ | No sub-folder convention | Purpose-based sub-folders |
| docs/ chaos: task packs, domains, releases mixed | No doc hierarchy | Type-based sub-folder tree |
| Chinese + English folder names mixed | No naming convention | Enforce ASCII for code, UTF-8 for docs |

**Principle**: The folder structure must match the project's architecture archetype and scale. A rapid prototype should not have the same structure as an enterprise system.

---

## 2. Root-Level Standard | 根目录标准

### What goes in root (project root)
```
{project}/
  AGENTS.md              ← Required: methodology constitution
  .editorconfig          ← Required: encoding/line-ending rules
  .gitignore             ← Required
  README.md              ← Project overview (NOT methodology docs)
  package.json           ← If Node.js project
  pom.xml / build.gradle ← If Java project (or under backend/)
```

### What NEVER goes in root
- Scripts → `scripts/`
- SQL → `database/`
- Documentation → `docs/`
- Temporary files → `temp/` or `tmp/`
- Configuration → `config/` or `backend/config/`
- Logs → `logs/`

## 4. Database Folder Standard | 数据库文件夹标准

**Problem from GERP**: `database/` had ~300 loose SQL files. Impossible to find anything.

**Solution**: Purpose-based sub-folders.

```
database/
  migrations/            ← V{NNN}__description.sql (Flyway format)
    V001__init_schema.sql
    V002__add_user_table.sql
  seed/                  ← Seed/test data, one file per module
    seed_users.sql
    seed_products.sql
  schema/                ← Reference documentation (generated, not hand-written)
    schema_v1.0.md
  backup/                ← Backup scripts
    backup_full.sql
  tools/                 ← Schema comparison, sync utilities
    sync_schema.py
```

**Naming rules**:
- Migration files: `V{NNN}__{description}.sql` (Flyway standard)
- Seed files: `seed_{module}.sql`
- No loose SQL files in database/ root — **auto-fail audit**

## 5. Docs Sub-Folder Hierarchy | 文档子文件夹层级

**Problem from GERP**: `docs/` had flat folders like "docs/报表规划任务包_20260328" mixed with "docs/财务模块". No way to distinguish task packs from domain docs from release docs.

**Solution**: Type-first, then topic.

```
docs/
  _templates/            ← Templates (never edit directly)
    架构决策记录/
    每日调研回写/
    业务流程全案/
    测试验收报告/
    部署运维手册/
    全项目总控/

  架构决策记录/           ← One file per decision
    ADR-001_多租户架构选型.md
    ADR-002_原子服务拆分策略.md

  每日调研回写/           ← One file per day
    2026-06-17_可行性分析.md
    2026-06-18_技术选型.md

  业务流程全案/           ← One sub-folder per process
    采购流程/
      00_总览.md
      01_采购订单.md
      02_采购入库.md
    销售流程/
      00_总览.md

  测试验收报告/           ← One file per test round
    2026-06_R1_核心流程测试.md
    2026-06_R2_回归测试.md

  部署运维手册/           ← One file per environment
    dev_部署指南.md
    prod_部署指南.md
    回滚流程.md

  全项目总控/             ← Master control
    MASTER_INDEX.md       ← Auto-generated skill/schema/doc index
    PROJECT_STATUS.md     ← Current phase, active tasks, blockers
```

**Rules**:
- Task packs go under their domain folder, not as loose `docs/{name}_任务包_{date}/`
- Release docs go under `docs/发布闭环/` (if needed) or `docs/部署运维手册/`
- No loose `.md` files in `docs/` root

## 6. Naming Convention | 命名规范

| Type | Convention | Example |
|---|---|---|
| Folders (code) | kebab-case, ASCII only | `auth-service/`, `user-profile/` |
| Folders (docs) | Chinese or kebab-case (consistent per project) | `架构决策记录/` or `adr/` |
| Files (code) | PascalCase for classes, camelCase for others | `UserService.java`, `formatDate.ts` |
| Files (docs) | `{date}_{topic}.md` or `{prefix}-{NNN}_{topic}.md` | `2026-06-17_部署.md`, `ADR-001_架构.md` |
| SQL migrations | `V{NNN}__{description}.sql` | `V001__init_schema.sql` |

**Enforcement**: `ai-component-standardizer` audits folder naming on project init.

## 7. What GERP Got Right | GERP 做得好的

| Pattern | Why It Works |
|---|---|
| `backend/gerp-modules/` with domain sub-folders | Clear module boundaries, independent builds |
| `frontend/` with platform sub-projects | Web, mobile, mini-program all independent |
| `scripts/` with type sub-folders | `bat/`, `ps1/`, `py/` — find by tool type |
| `docker/` with env sub-folders | `dev/`, `prod/`, `test-server/` — clear purpose |
| raw skill archive preserved separately | Keep battle-tested source material while exposing only cleaned `skills/` as the official skill root |

## 8. What GERP Got Wrong (and We Fixed) | GERP 的问题

| Problem | Our Fix |
|---|---|
| 55 loose files in root | Only AGENTS.md + .editorconfig + build files |
| ~300 loose SQL in database/ | `migrations/`, `seed/`, `schema/`, `backup/` sub-folders |
| docs/ flat with no hierarchy | Type → Domain → Date structure |
| Chinese + English mixed folders | Per-project consistency rule |
| Multiple AI tool configs (6 tools) | Standardize source on `skills/`, then deploy to tool adapter folders with `tools/deploy.ps1` |
