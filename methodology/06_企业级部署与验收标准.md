## 1. Why Deployment Gates? | 为什么需要部署门禁？

AI-generated code passes through the same deployment pipeline as human-written code. Without explicit gates, AI can deploy:

- Code that compiles but fails at runtime
- Code that passes tests but breaks in production
- SQL migrations that work locally but fail on remote servers
- Configuration that uses wrong environment variables

**The gate system treats AI-generated code exactly like human code — no exceptions.**

---

## 2. Pre-Deployment Gates | 部署前门禁

### Gate 1: Build Verification
`ash
# Must pass with zero errors
mvn clean compile -DskipTests    # or equivalent
npm run build                    # frontend
`

### Gate 2: Unit Test Gate
`ash
# All tests must pass
mvn test                         # backend
npm run test                     # frontend
`

### Gate 3: Database Schema Gate (Template — customize paths to your project)
`ash
# Zero schema drift between template and all instances
scripts/check-schema-sync.sh

# Required tables and columns exist in all schemas
scripts/check-database-baseline.sh
`

### Gate 4: Code Quality Gate
`ash
# Linting passes
npm run lint                     # frontend
# Static analysis passes
mvn verify                       # backend
`

### Gate 5: Security Scan
`ash
# No known vulnerabilities in dependencies
npm audit --production           # frontend
mvn dependency-check:check       # backend
`

## 4. Acceptance Criteria | 验收标准

### Per-Feature Acceptance

| Criterion | Evidence Required |
|---|---|
| **Functional completeness** | All acceptance test cases pass |
| **Error handling** | Invalid inputs return correct error codes |
| **Edge cases** | Null, empty, max-length, special chars tested |
| **Performance** | Response time within SLA (e.g., <500ms for CRUD) |
| **Security** | Endpoint requires authentication, authorization checked |
| **Documentation** | API docs updated, ADR if architecture changed |

### Per-Release Acceptance

| Criterion | Gate |
|---|---|
| **All pre-deployment gates green** | Build, test, schema, lint, security |
| **Staging smoke test passes** | Health, login, core CRUD |
| **Zero P0/P1 bugs open** | Bug tracker verified |
| **Rollback plan documented** | Written and tested |
| **Release notes published** | Changelog + migration guide |

## 5. Rollback Procedure | 回滚流程

Every deployment must have a **tested rollback path**:

`ash
# 1. Stop current deployment
systemctl stop app-service

# 2. Restore previous artifact
cp /backups/app-v{previous}.jar /deploy/app.jar

# 3. Rollback DB migrations (if reversible)
flyway undo

# 4. Start previous version
systemctl start app-service

# 5. Verify health
curl http://localhost:8080/actuator/health
`

**Rule**: If rollback takes longer than 5 minutes, the deployment was not properly gated.

## 6. Monitoring & Alerts | 监控与告警

After deployment, monitor for:

| Metric | Alert Threshold | Action |
|---|---|---|
| Error rate | >1% of requests | Investigate immediately |
| Response time | >2x baseline | Check for regressions |
| Memory usage | >80% of max | Potential memory leak |
| DB connections | >80% of pool | Connection leak check |
| Health endpoint | Not UP | Service restart |

---

## 7. AI-Specific Deployment Rules | AI特有部署规则

| Rule | Reason |
|---|---|
| **AI must not deploy directly to production** | Human review required for prod deployments |
| **AI-generated SQL must be reviewed** | DDL mistakes can cause data loss |
| **AI must run pre-deployment gates before claiming "done"** | Prevents "it compiled on my machine" |
| **AI must produce rollback scripts for every migration** | No irreversible changes without explicit approval |
