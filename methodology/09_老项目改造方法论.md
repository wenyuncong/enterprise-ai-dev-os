> **Part of**: Enterprise-Grade Full AI Development Methodology

## 1. Why Brownfield ≠ Greenfield | 为什么老项目不能当新项目做？

Greenfield assumes an empty canvas. Brownfield is surgery on a living body. The difference:

| | Greenfield | Brownfield |
|---|---|---|
| **Starting point** | Empty directory | Existing codebase with history |
| **Architecture** | Can design from scratch | Must work within existing architecture |
| **Risk tolerance** | High (nothing to break) | Low (users/data depend on it) |
| **Pattern source** | Methodology provides patterns | Project provides patterns (extract, don''t impose) |
| **Skill evolution** | Skills grow from new patterns | Skills grow from EXTRACTED patterns |
| **Speed** | Fast (no constraints) | Variable (depends on code quality) |

**The golden rule of brownfield**: **The existing project is the teacher. Your methodology is the student.** Extract patterns first, then decide what to keep, fix, or replace.

## 2. Intervention Spectrum | 干预深度分级

Not all brownfield work needs deep renovation. Classify the intervention level first.

```
┌─────────────────────────────────────────────────────────────┐
│                  INTERVENTION SPECTRUM                       │
├────────┬────────┬──────────┬──────────┬────────────────────┤
│ Level 1│Level 2 │ Level 3  │ Level 4  │ Level 5            │
│ Quick  │Feature │ Module   │ Deep     │ Takeover           │
│ Fix    │Addition│ Upgrade  │ Renovation│ (Semi-finished)   │
├────────┼────────┼──────────┼──────────┼────────────────────┤
│ 1-2 hrs│1-5 days│1-4 weeks │1-6 months│ Ongoing            │
│ Bug fix│New feat│Refactor  │Restructure│Inherit + finish    │
└────────┴────────┴──────────┴──────────┴────────────────────┘
```

### Level 1: Quick Fix (1-2 hours)
**When**: Bug fix, minor text change, simple feature addition
**Approach**: 
- Read the affected file + its direct dependencies
- Match existing code style exactly
- Don't refactor anything unrelated
- Test only the changed path
**Gate**: Fix works, nothing else broke

### Level 2: Feature Addition (1-5 days)
**When**: Adding a new module/feature within existing architecture
**Approach**:
- Study 2-3 similar existing modules for patterns
- Copy their structure, naming, patterns exactly
- No architecture changes
**Gate**: New feature works, matches existing patterns

### Level 3: Module Upgrade (1-4 weeks)
**When**: Refactoring one module while keeping interfaces stable
**Approach**:
- Deep audit of the module (code, DB, API, tests)
- Extract patterns before changing anything
- Refactor internals, keep public API stable
- Update skills with extracted patterns
**Gate**: Module quality improved, no regression in dependent modules

### Level 4: Deep Renovation (1-6 months)
**When**: Restructuring architecture, migrating tech stack, multi-module refactor
**Approach**:
- Full project audit (all modules, all layers)
- Pattern extraction → skill creation/update
- Architecture decision records for each change
- Incremental: one module at a time, keep system running
**Gate**: Each module passes independently, system stays operational

### Level 5: Takeover (ongoing)
**When**: Inheriting semi-finished work from another team/developer
**Approach**:
- Full audit: what works, what doesn't, what's missing
- Document current state honestly (not what was "supposed" to be)
- Classify work items: working / buggy / unfinished / absent
- Build a "finish order" based on dependency, not wishlist
**Gate**: Honest state documented, realistic finish plan created

## 3. The 6-Phase Brownfield Workflow | 六阶段改造流程

```
PHASE 1: AUDIT        → What exists? What''s the current state?
PHASE 2: CLASSIFY     → Which intervention level? What quality target?
PHASE 3: EXTRACT      → What patterns to keep? What to fix? → Feed skills
PHASE 4: PLAN         → What to do, in what order, with what risk?
PHASE 5: EXECUTE      → Surgical changes, respect existing patterns
PHASE 6: EVOLVE       → Feed extracted patterns back to skills
```

---

### Phase 1: AUDIT — Analyze Current State | 现状分析

**DO NOT write a single line of code in this phase.** Audit only.

#### 1.1 Project Inventory
```markdown
## Project Inventory
| Item | Count / Status |
|---|---|
| Total files | [N] |
| Languages | [List] |
| Framework(s) | [List with versions] |
| Database tables | [N] |
| API endpoints | [N] (count from routes) |
| Test files | [N] |
| Test coverage | [% estimate] |
| Last commit | [Date] |
| Dependencies | [Count, any flagged as vulnerable] |
```

#### 1.2 Code Quality Scan
```markdown
## Code Quality Findings
| Issue | Severity | Files Affected | Recommendation |
|---|---|---|---|
| No tests | P0 | All modules | Add tests for critical paths first |
| Mixed naming conventions | P1 | ~40% of files | Standardize gradually |
| Duplicate logic across services | P1 | Auth, Validation | Extract to shared module |
| Hard-coded config values | P1 | ~15 files | Move to config/env |
| Missing error handling | P2 | Several endpoints | Add try/catch, error responses |
```

#### 1.3 Architecture Mapping
```markdown
## Architecture Map
### Backend
- Services: [List with responsibilities]
- Service boundaries: [Clear or fuzzy?]
- Cross-service dependencies: [List]
- Database: [Schema structure, tenant model]

### Frontend  
- Pages: [Count by type: list/document/report/dashboard]
- Component reuse: [High/Medium/Low]
- State management: [Pattern used]
- API pattern: [Direct calls or service layer?]

### Pain Points (from code + team feedback)
1. [Pain point 1]
2. [Pain point 2]
```

#### 1.4 Risk Assessment
```markdown
## Risk Assessment
| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Breaking existing functionality | High | High | Comprehensive regression tests before refactor |
| Database migration failure | Medium | Critical | Backup first, dry-run migration, rollback plan |
| Performance degradation | Low | Medium | Benchmark before/after |
```

### Phase 2: CLASSIFY — Determine Intervention Level | 干预分级

Based on audit findings + user goals:

| Question | Determines |
|---|---|
| What does the user actually want? | Fix one thing vs. renovate everything |
| How urgent is it? | Timebox for the intervention |
| What's the code quality baseline? | Realistic scope of what can be achieved |
| Are there users depending on this? | Risk tolerance level |
| Is this semi-finished work? | Takeover vs. renovation |

**Output**: Level 1-5 classification with explicit scope boundary.

### Phase 3: EXTRACT — Pattern Mining | 模式提炼

**This is where skills evolve from legacy code.** Every project has patterns — some good, some bad. Extract both.

#### 3.1 Good Patterns to Keep and Formalize
```markdown
## Extracted Patterns → Skill Updates

### Pattern: [Name]
- **Found in**: [File/Module]
- **What it does**: [Description]
- **Why it works**: [Analysis]
- **Skill to update**: [Skill name]
- **Update**: [What to add to the skill]
```

#### 3.2 Bad Patterns to Fix (Anti-Patterns)
```markdown
## Anti-Patterns Found → Fix Plan

### Anti-Pattern: [Name]
- **Found in**: [Files affected]
- **What's wrong**: [Description of the problem]
- **Impact**: [What it causes — bugs, perf issues, maintenance pain]
- **Fix approach**: [How to refactor]
- **Priority**: [P0/P1/P2]
```

#### 3.3 Extraction Output Feeds
```
Extracted patterns → ai-skill-evolver → Update/create skills
Extracted anti-patterns → ai-component-standardizer → Add to forbidden patterns
Project conventions → ai-component-standardizer → Template updates
Architecture map → ai-architect-governor → ADR
Library inventory → ai-library-first → Library catalog update
```

### Phase 4: PLAN — Surgical Plan | 改造计划

**Brownfield plans are about risk management, not ambition.**

```markdown
## Brownfield Intervention Plan

### Scope
- **Intervention Level**: [1-5]
- **In scope**: [Explicit file list or module list]
- **Out of scope**: [Explicit exclusion — what we will NOT touch]
- **Reason**: [Why this scope — e.g., "Only the purchase module; inventory is stable"]

### Execution Order
| Batch | Description | Dependencies | Risk | Time Estimate |
|---|---|---|---|---|
| 1 | Add tests for critical paths | None | Low | 2 days |
| 2 | Extract duplicate auth logic | Batch 1 | Medium | 3 days |
| 3 | Refactor purchase service | Batch 1,2 | High | 5 days |

### Rollback Plan
- [How to undo each batch if something breaks]
- [Database backup strategy]
- [Which tests must pass before proceeding to next batch]
```

---

### Phase 5: EXECUTE — Surgical Changes | 外科手术式修改

**Rules for modifying legacy code:**

| # | Rule | Rationale |
|---|---|---|
| 1 | **Match existing style, even if you disagree** | Consistency > correctness of style preference |
| 2 | **One change type per commit** | Refactor OR feature, never both in one commit |
| 3 | **Add tests before changing behavior** | Tests are the safety net for refactoring |
| 4 | **Keep the system running** | Never break main branch; use feature branches |
| 5 | **Don't fix what isn't broken** | If it works and isn't in scope, don't touch it |
| 6 | **Database changes get extra caution** | Backup + dry-run + rollback script + verify |
| 7 | **Interface stability over internal beauty** | Public APIs stay compatible; refactor internals |

### Level-Specific Execution Rules

| Level | Execution Rules |
|---|---|
| 1 (Quick Fix) | Read only the affected file + direct deps. Copy existing style exactly. |
| 2 (Feature) | Copy 2-3 similar modules. Same structure, naming, patterns. |
| 3 (Module Upgrade) | Tests first. Refactor internals. Keep public API. One module at a time. |
| 4 (Deep Renovation) | Full audit → ADR per change → Incremental batches → Keep system running. |
| 5 (Takeover) | Document real state first. Prioritize by dependency. No "nice to have" until basics work. |

## 4. Semi-Finished / Takeover Projects | 半成品/接手项目

### Red Flags in Takeover Projects
| Red Flag | What to Do |
|---|---|
| No README or documentation | Write one yourself during audit (Phase 1) |
| No tests | Add smoke tests before ANY changes |
| Mixed languages/frameworks | Document what's used where; don't add more |
| Dead code / commented-out blocks | Note in audit, remove only in dedicated cleanup batch |
| "It almost works" claims | Verify every claim independently; trust only what you see running |
| No database migrations | Document schema manually before any DB changes |

### Takeover Audit Template
```markdown
## Takeover Audit: [Project Name]

### What Works (verified by running)
| Feature | Status | Notes |
|---|---|---|
| User login | ✅ Working | |
| Dashboard | ⚠️ Partial | 3 of 5 widgets render |
| Report export | ❌ Broken | 500 error |

### What's Missing (not implemented)
| Feature | Priority | Dependencies |
|---|---|---|
| Email notifications | P1 | SMTP config |
| Mobile responsive | P2 | CSS refactor |

### What's Buggy
| Bug | Severity | Reproduction Steps |
|---|---|---|
| Date picker shows wrong timezone | P1 | Select date, observe UTC offset |

### Finish Order (dependency-based, not wishlist)
1. Fix critical bugs (P0)
2. Complete partially-working features
3. Add missing must-have features
4. Add tests for all of the above
5. Only then: nice-to-have features
```

## 5. Common Brownfield Traps (from GERP Experience) | 常见陷阱

| Trap | What Happens | Prevention |
|---|---|---|
| **"Let's just rewrite it"** | 6 months later, the new version has different bugs and the old version still runs | Never rewrite; renovate incrementally |
| **"I'll fix this while I'm here"** | One bug fix becomes a 3-day refactor with 5 side effects | Surgical focus: fix only what's in scope |
| **Imposing new patterns** | Team has used Pattern A for 2 years; AI imposes Pattern B → inconsistency chaos | Extract existing patterns, don't impose new ones |
| **Skipping the audit** | Start coding immediately → break something that "looked wrong" but was intentional | Audit first, always |
| **Adding libraries without checking existing ones** | Project already uses Chart.js; AI adds ECharts "because it's better" → bloat | Check ai-library-first, respect existing choices |
| **Testing only the new code** | New feature works; old feature breaks → regression | Full regression test on every change |

## 5b. Additional Brownfield Scenarios | 其他未考虑场景

### Scenario: Original Team Unavailable | 原始团队不在

**Indicators**: No one knows why certain decisions were made. "It's always been like that."

```
Handling "no original team" projects:

1. TREAT CODE AS THE ONLY DOCUMENTATION
   - Comments may be outdated. Code is the truth.
   - Git blame is your best historian (who changed what, when)

2. REVERSE-ENGINEER BUSINESS RULES FROM CODE
   - "Why is discount capped at 15%?" → Find the constant/method
   - Don't guess. The answer is in the code somewhere.

3. ADD TESTS AS YOU UNDERSTAND
   - Each time you figure out "what this does", add a test
   - Tests become the new documentation for the next developer

4. DON'T ASSUME INTENT
   - "This looks wrong" ≠ "This IS wrong"
   - It might be a workaround for a condition that no longer exists
   - Or it might be intentional for a reason you haven't discovered yet

5. BUILD A "BUS FACTOR" DOCUMENT
   - Which modules have zero surviving knowledge?
   - Prioritize documenting those first
```

### Scenario: Data Migration Required | 需要数据迁移

**When**: Old system has live data that must move to new/upgraded system.

```
Data Migration Checklist:

[ ] Audit source data quality FIRST
    - Null values? Duplicates? Inconsistent formats?
    - Fix at source before migration, not after

[ ] Map old schema → new schema
    - One mapping document per table
    - Flag: direct map / needs transform / can't migrate

[ ] Write migration script
    - Idempotent (can run multiple times safely)
    - Has dry-run mode
    - Logs every row changed

[ ] Test on copy of production data
    - Never test migration on live data
    - Verify row counts match (source vs target)

[ ] Plan rollback
    - If migration fails halfway, how to restore?
    - Backup + transaction + checkpoint strategy

[ ] Schedule migration window
    - During lowest traffic
    - With monitoring active
    - With rollback team on standby
```

### Scenario: Environment Drift | 环境不一致

**When**: "It works on my machine" but fails in staging/production. Config files have diverged.

```
Environment Drift Audit:

[ ] Compare configs across environments
    diff dev/application.yml staging/application.yml
    diff staging/application.yml prod/application.yml

[ ] Check for:
    - Different dependency versions (package.json vs package-lock.json)
    - Different Node/Java/Python versions
    - Different database versions
    - Hard-coded paths that only exist on one machine
    - Environment variables set manually (not in config)

[ ] Standardize:
    - One config template + env-specific overrides
    - Docker for environment consistency
    - .env.example committed, .env git-ignored

[ ] Verify:
    - Fresh clone → one command to run locally
    - README must have exact setup steps
    - If setup takes > 10 minutes, something is wrong
```

### Scenario: When to Walk Away | 何时放弃重做

Sometimes the kindest thing is to admit a project needs a fresh start.

```
Decision Criteria: Rebuild vs Renovate

| Factor | Weight | Score 1-5 |
|---|---|---|
| Tests exist and pass | 20% | [SCORE] |
| Dependencies are current (< 12 months old) | 15% | [SCORE] |
| Architecture is coherent (not patchwork) | 20% | [SCORE] |
| Team understands the codebase | 15% | [SCORE] |
| Core features work correctly | 20% | [SCORE] |
| Security baseline is met | 10% | [SCORE] |

Weighted Score < 2.5 → STRONGLY CONSIDER REBUILD
Weighted Score 2.5-3.5 → RENOVATE WITH CAUTION
Weighted Score > 3.5 → RENOVATE (project is salvageable)
```

**Rebuild rules**:
- Extract ALL business rules from old system first (they are the real asset)
- Keep the database schema if possible (data > code)
- Rebuild incrementally, keep old system running until new is proven
- This is NOT greenfield — you carry the business rules with you
